### 1. Context:

ENG1013's Traffic Light System Project (TLSP) focused on working in a small team to reproduce and simulate a real-world, UK-based traffic light system using the unit's provided electrical kits. The project aimed to address the complex traffic network in the UK while providing a challenging opportunity to teamwork collaboration and conflict resolution. TLSP was completed within 3 months.

The system comprised five subsystems, each with required, general, and integration features. My responsibility was to design and build Subsystem 4, an over-height detection system at the tunnel entrance, consisting of ultrasonic sensors (US3/US4), a traffic light (TL3), and two sets of red warning lights (WL2).

### 2. Planning:

**My subsystem had the following features:**

- Upon detection of an overweight vehicle by US3, TL3 turns from green to red immediately. The system resets only when no over-height vehicle is detected.
- US4 verifies US3 readings within an acceptable error range (3 cm).
- On startup, the user is prompted for an overweight limit (default 4.0 m), used as a variable throughout the subsystem.
- WL2 flashing pattern XOXO/OXOX with flashing frequency generated by a 556 timer.

**Total time:**

The following tasks include the estimated time for researching, testing, redesigning, and rebuilding over the course of 3 months.

- Circuit design (Draw.io): 10 hours
- Python script: 15 hours
- Physical circuit building: 20 hours

**Resource Allocation:**

- Hardware: HC-SR04 sensors (×2), 556 timer, resistors (10 kΩ, 330 Ω), capacitor (10 µF), red/green LEDs, BS170 NMOS transistor, breadboard, solid-core wire
- Software: VSCode, Python 3.13 with Pymata4, Arduino library FirmataExpress

**Risk Assessment:**

- LEDs blowing up: Likely. This was mitigated by pre-calculating resistance values from the schematic sketches before testing on a physical circuit.
- Overheating the 556 timer: Low. Input voltage was a maximum of 5 V.
- Damage to the capacitor: Low. I ensured capacitor's terminals were properly connected and capacitance value was not below 10 µF.
- Wire stripping injury: High. Wearing lab-standard glasses helped prevent wires from causing eye injuries.

### 3. Building:

**Hardware:**

The WL2 warning light circuit required alternating flashing of four red LEDs in an XOXO/OXOX pattern. I used Timer B of the 556 timer (pins 7-14) configured in astable mode. The timing components were:

- $R_3$ = 10 kΩ ($R_A$)
- $R_4$ + $R_5$ = 20 kΩ ($R_B$)
- $C_1$ = 10 µF

**Period calculation:**

T = 0.693 × ($R_A$ + 2$R_B$) × C = 0.693 × (10000 + 40000) × 10 × $10^{-6}$ = 0.3465 s

**Software:**

The Python script sequentially polled US3 and US4, measured distances, computed the absolute error, and controlled TL3 and WL2 based on the configured threshold. Key design decisions:

1. Sequential polling to avoid ultrasonic interference
2. 50ms delay between sensor readings for stable measurements
3. Error range check before triggering red lights
4. On KeyboardInterrupt, all pins set LOW before `board.shutdown()`

```python
if distanceUS3 < overLC and errorRange <= 3:
    board.digital_write(greenTL3, 0)
    board.digital_write(redWL2, 1)
    board.digital_write(redTL3, 1)
elif distanceUS3 >= overLC:
    board.digital_write(redTL3, 0)
    board.digital_write(redWL2, 0)
    board.digital_write(greenTL3, 1)
```

Note that distanceUS3 is measured between the top of the vehicle and the ultrasonic sensor located above it, and overLC after sizing becomes the minimum gap.

### 5. Challenges:

In WL2 subcircuit, a 556 timer was intended to generate flashing lights in patterns XOXO/OXOX alternately using four red LEDs, which were arranged in two parallel pairs across two separate branches. However, upon power-up, all LEDs illuminated simultaneously at an approximately equal brightness.

**Investigation:**

Checked the values of resistors and capacitor connected to the 556 timer:

- For $R_A$ = $R_3$ and $R_B$ = $R_4$ + $R_5$, their values were $R_3$ = $R_4$ = $R_5$ = 10 kΩ, and $C_1$ = 10 μF, therefore period = 0.3465 s, which was sufficient to produce visible flashing.
- The 556 timer section was reviewed and confirmed that it was correctly connected to the positive and negative rails.
- The WL2 section was analysed, and it was identified that both parallel branches had an identical configuration. This was therefore the root cause.

**Root Cause:**  

Each branch of WL2 consisted of two identically configured "LED + resistor" strings connected in parallel. Each red LED was labelled with D- and each resistor (330 Ω) was labelled with R-.

- Branch 1: Output → ($D_4$ + $R_7$) // ($D_6$ + $R_9$) → GND
- Branch 2: Output → ($D_3$ + $R_6$) // ($D_5$ + $R_8$) → GND

During charging of the capacitor ($C_1$), the lower comparator (inside Timer B of 556 timer) output HIGH to input S of flip-flop, thus state of Q = HIGH, signaling to the Output Driver behind the Output pin to open the path for the current.

Since both branches shared the same nodes at both ends (between Output pin and GND), all the LEDs illuminated simultaneously.

**Fix #1:**

1. **Swapped the positions of each of the LEDs and their adjacent resistors in branch 2.**
2. **Disconnected branch 2 to GND and reconnected to positive rail (5 V).**

As the voltage in $C_1$ crossed 2/3 of 5 V, discharging started, Q went LOW, signaling the Output Driver to block the path between 5V and Output, and open the path between GND and Output. Current then flowed from the 5 V rail into Branch 2, turning on its LEDs, then continued into the Output pin to ground. Branch 1 was turned off since both sides equaled 0 V; thus, current = 0 A.

However, since the internal transistor inside the Output Driver still needed a small voltage to turn on and pulled the Output pin low, there was a voltage residue that remained relative to ground, leading to a dim light in $D_4$ and $D_6$.

**Fix #2:**

1. **Connected an NMOS transistor (BS170) to branch 1**.

This solved the light dimming issue by using the transistor's threshold voltage (~2.0 V) to block the current path; therefore, when the node voltages in both sides were identical (5 V), the voltage difference equaled 0 V, current was 0 A. As a result, the $D_4$ and $D_6$ turned off.

**Lesson:**

- LEDs only blink alternately when they are configured in such a way that the flow of current is opposite.
- Adding an NMOS transistor can block the current path if the gate-source voltage is below the threshold voltage.

### 6. Reflection

**What went well:**

- Systematic debugging: The parallel LED issue was resolved through methodical investigation by checking timing components first, then timer connections, then branch configuration.
- Clean code structure: The Python script handles errors smoothly (ValueError, TypeError) and shuts down on KeyboardInterrupt as expected.
- Hardware understanding: Building the 556 timer circuit from first principles deepened my understanding of astable multivibrators, comparators, and flip-flops.
- Time management: Required features were completed ahead of schedule, allowing time for general features and debugging.

**What I would do differently:**

1. Test WL2 subcircuit by simulating it first. Using an online 3D modelling program such as TinkerCad would have revealed the oscillation frequency earlier.
2. Review branch configurations carefully before implementation.

**What I learned:**

- The internal architecture of 555 timer (in 556), which includes comparators, flip-flop, discharge transistor, and how it generates square waves.
- How to calculate:
	- Oscillation period: T = 0.693 × ($R_A$ + 2$R_B$) × C
	- Oscillation frequency: f = 1.44 / (($R_A$ + 2$R_B$) × C)
- The importance of threshold voltages in MOSFETs for blocking unwanted current
- Sequential polling for ultrasonic sensors to avoid interference
- Clean error handling in embedded Python applications

**How this influenced me:**

This project reinforced that engineering is as much about debugging as it is about building functional systems. The parallel LED issue taught me to plan and test before incorporating WL2 into Subsystem 4, as well as the patience to look into details and analyse imperfections logically from a more skeptical perspective.
